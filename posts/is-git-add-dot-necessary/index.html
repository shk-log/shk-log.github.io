매번 치는 git add . 꼭 필요한 절차인가? | Staging Area의 재발견

December 12, 2025 · 3 min · JSON

git 블로그를 작성하면서 Shell Script를 사용하지 않고 굳이 수동으로 add, commit을 입력하는 방식을 고수하고 있다.(이유: Git blog를 시작하는 이유) 그러다보니 의문이 생겼다. 항상 습관처럼 git commit과 git push를 하기 전에 git add .를 치고 있는데, 이게 정말 매번 필요한 절차일까? 직전 글을 올리면서 h태그 문제 때문에 Git 명령어를 반복해서 치다가 든 생각이다.

공통 코드는 묶어 사용했던 경험#

누가 시키지 않았는데 스스로 공통코드를 묶어서 비효율을 줄이겠다고 생각하고 실행했던 일은 최종 프로젝트 때 웹소켓 설정했을 때의 일이다.

상태가 변할때마다 알림이 작동해서 교수에서 학생쪽으로 메시지를 보내는데 알림 받는 아이디, 내용, 알림 클릭시 이동주소 만 변하고 나머지는 공통된다는 것을 인지했다. 그렇게 알림이 3번 반복된다고 할때 “아 이건 비지니스 로직 내부에서 3개 변수를 인자로 받는 매서드를 하나 만들어야 겠다"는 생각이 들었고, 그렇게 코드를 작성했다.

코드는 간결해지고, 오류는 줄어들며(한개의 알림만 제대로 가면 나머지 알림에 문제가 생겼을때 적어도 공통코드 문제는 아니라는 판단이 들것이다), 가독성은 늘어났다. 알림을 3번 사용할 때는 코드를 2/3 줄인 셈이지만, 100번 사용한다면 99%를 줄인 셈이 된다. 꽤나 효율적인 리팩토링이었다.

이런 경험이 있다 보니 git add . 를 반복하는 것이 뭔가 비효율적으로 보였다. 그래도 존재하는 이유가 있겠지. 존재의 이유를 파악하기 위해 Git 공식 문서를 살펴봤다.

Git 3개의 영역과 4가지 상태#

Git은 파일을 관리하기 위해 3가지 영역(Working Directory, Staging Area, Repository)를 사용한다. 그리고 파일이 어느 영역에 위치하느냐에 따라 4가지 상태 Untracked, Modified, Staged, Committed 로 구분한다.

출처 : Git book | Git the Tree States

이해한 바를 적어본다면, 일단 우리가 작업한 것은 Working Directory에 있다. git이 처음 보는 새로운 파일이면 Untracked 상태(추적하지 않는 상태)이고, 이미 git이 추적하고 있었는데 수정된 거면 Modified 상태가 된다. 이걸 git add . 를 할 경우 해당 내 현재 디렉토리의 하위에있는 모든 파일이 Staging Area 에 올라가면서 Staged 상태가 된다. 여기서 commit을 하면 데이터가 로컬 데이터 베이스에 안전한 상태로 저장되며 Committed 상태가 된다.

현재 내 파일의 상태는 git status 명령어로 확인할 수 있다.

Staging Area 가 필요한 이유 = git add 가 필요한 이유#

만약 git add . 하는 경우만 있다면 Staging Area는 필요가 없다. Git 은 형상관리 도구이다. 블로그 글 작성시 하나의 문단엔 하나의 내용을 담아야 하듯, 하나의 커밋엔 하나의 변경사항만 담는것이 기억하기도 나중에 추적관리하기에도 좋을 것이다. 오늘 하루 동시에 여러 작업을 했다면 논리적 단위의 커밋을 담기 위해 git add . 이 아닌 git add 파일명 or 폴더명을 적고 여러번 커밋을 할 필요가 있다. 당장은 귀찮아 보이는 행위일지라도 이렇게 정리해두면 나중에 추적관리하기가 좋을 수 밖에 없다.

만약 git add . 만 하는 상황이라면#

나처럼 push 전에 글 수정사항을 확인하고 수정후 다시 add와 commit을 해야 하는 상황이라면 git commit -am "message" 이렇게 -a(all) 명령어를 사용해서 Staging Area를 생략(git add명령어 생략)하고 바로 커밋할 수도 있다. 다만 공식 문서에 따르면 이럴경우 Modified 상태만 커밋되고 Git이 추적하지 않은 새로 생성한 파일(Untracked) 상태의 파일은 커밋 되지 않으니 주의해야 한다.
