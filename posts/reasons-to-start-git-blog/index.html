Git 블로그를 시작하는 이유

December 8, 2025 · 2 min · JSON

나는 블로거다. 협찬을 받아본 적은 없지만 구글 애드센스로 수익 실현을 다년간 했었으니 나름 전문 블로거라 할 수 있다. 개발자가 되기 위해 대덕인재개발원에서 교육을 받았고 이제 그 내용을 나름 정리해보려고 하는데 Git 블로그를 선택했다.

Git을 잘하고 싶어서#

첫 번째 이유는 Git을 잘하고 싶어서다. 교육 기간, 특강 중 어떤 대표님께서 “파이썬을 잘하고 싶으면 파이썬 프로젝트 하기를 바랐다” 라고 말씀하셨다. 사람마다 배우는 것이 다를 수 있지만 적어도 난 대표님의 말에 크게 공감했다. 실제로 교육 기간 동안 프로젝트를 진행할 때, 그것에 몰입할 때 프로그램에 대한 이해도가 높아지는 게 느껴졌다.

Git도 잘하고 싶다. 깃을 잘하려면 깃 명령어를 많이 써봐야 할 것이다. 깃 블로그에서 저장은 git commit이고, 발행은 git push다. 글 하나하나 작성할 때마다, 오타 하나 고칠 때도 명령어를 쳐야 하니 블로그를 작성하면 할수록 Git 명령어가 손가락에 문신처럼 새겨질 거라 믿는다.

개발자스러운 글을 쓰고 싶어서#

나름 10년 블로거다. 하지만 내 글이 개발자스럽지 못하다는 것을 느꼈다. 위의 “Git잘하고 싶어서” 부분을 보면 최종프로젝트를 통해 이해도가 높아졌는데 ‘무엇이 어떻게 왜’ 높아졌는지가 안 적혀 있다.

이 문제는 이력서를 작성하면서 알았다. 이력서를 봐주신 선생님께서 내 이력서를 보시더니 “그래서 뭘 어떻게 했는데?” 가 안 나오고 내용이 두루뭉술하다고 지적하셨다. 사실 그게 어찌 보면 온라인 마케팅의 핵심일지도 모른다. 내용은 없지만 글자 수를 채우되 사람들로 하여금 계속 읽게 하고 체류시간을 늘리는 것 말이다.

하지만 개발자의 길을 걷는다면 더 이상 예전 블로그 글 쓰기 방식을 고수해선 안 된다. 왜 그걸 했고, 어떤 문제가 있고, 어떻게 행동했고, 결과는 어땠으며, 그 경험을 통해 무엇을 배웠는지 직관적으로 읽혀야 한다.

README의 실패 경험#

최종 프로젝트 기간에 React의 초기 세팅을 내가 맡아서 했다. README.md를 작성하려고 하는데 굉장히 생소하게 느껴졌다. 급한 마음에 일단 나의 언어로 대략 적고 내용은 공유했지만, 효과적이지 못했다.

API 요청하는 파일만 따로 폴더에 모아 두자고 했는데 잘 지켜지지 않았다. 구두로 분명히 이야기했지만 잘 공유가 안 된 거다. README를 잘 적었다면 적어도 내가 공유했다는 근거가 명확히 남았을 것이다. 현업에 가면 문서화가 굉장히 중요하다는 생각이 들었다.

자유롭게 적더라도 **개발자들의 형식(Markdown)**에 맞는 글이 되어야 한다. Git 블로그를 마크다운으로 작성하는 습관이 이 부분에 조금이나마 도움이 될 거라 생각한다.

개발 경험은 자산이다#

실물이 없는 개발 세계에서 진정한 자산은 개발 경험인 것 같다. 중간 프로젝트 때 몰입하며 진하게 경험한 것들도 시간이 지나면 휘발되는 것이 있었다. Git 블로그에 기록하고 되새김질하며, 경험했던 그리고 경험할 프로젝트의 기억이 온전히 내 자산으로 남길 바란다.
